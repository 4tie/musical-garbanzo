HER is a profitable-strategy discovery and validation system.

It tries to find profitable trading strategies, but it never claims profitability without evidence.

Core rule:

AI suggests.
Backend validates.
Freqtrade tests.
HER decides.

# Task

Improve the Optimization configuration workflow.

The current Optimization flow requires the user to manually select:

* strategy
* trading pairs
* timeframe
* risk profile
* data days
* download missing data checkbox
* exchange
* max open trades
* stake amount
* notes

This is too manual and not smart enough.

The new flow should be:

User selects strategy
→ user selects risk profile
→ HER can auto-configure timeframe and timerange
→ user can still override timeframe and timerange
→ user can manually configure pairs OR load top pairs
→ missing data is downloaded automatically when needed
→ optimization runs with clear, safe configuration

Do not add mock data.
Do not add fake pairs.
Do not add fake charts.
Do not invent backend results.
Do not weaken safety rules.
Do not change validation thresholds.
Do not add approval/export/live trading.

---

# Main UX Goal

Make Optimization easier and smarter.

The user should not need to understand every technical detail before running optimization.

The UI should guide the user with smart defaults while still allowing manual override.

---

# Required Optimization Form Changes

## 1. Strategy Selection

Keep strategy selection.

The user must still select a strategy.

The selected strategy should drive:

* readiness status
* available actions
* validation warnings
* whether optimization can start

If the strategy is not ready, block optimization and show the readiness reason.

---

## 2. Trading Pairs

Replace the current rigid trading pair input with a better pair selection section.

The user should have two modes:

### Mode A: Manual pairs

User manually selects or enters trading pairs.

Examples:

* `BTC/USDT`
* `ETH/USDT`
* `SOL/USDT`

The UI should support:

* adding/removing pairs
* validation of empty list
* clear pair count
* warning if pair count is too high

### Mode B: Load top pairs

Add a button:

`Load Top 100 Pairs`

Behavior:

* It must load real pairs only.
* It must not use mock pairs.
* It must not hardcode fake data.
* It should use backend/exchange/pairlist data if available.
* If backend support is missing, add the smallest safe backend endpoint needed or document the missing support clearly.
* Pairs should be filtered by selected exchange and quote currency if supported.
* Default quote currency can be `USDT`.

Expected UX:

* Button: `Load Top 100 Pairs`
* Optional selector: `Top 20 / Top 50 / Top 100`
* Show loaded count.
* Show warning: “More pairs may increase runtime.”
* Allow user to remove pairs after loading.

If top pairs cannot be loaded, show a controlled error:

`Unable to load top pairs from real exchange data.`

Do not fallback to fake pairs.

---

## 3. Risk Profile Controls Timeframe Defaults

Risk profile should control default timeframe.

The user can still override timeframe manually.

Required behavior:

When user selects risk profile:

### Lower Risk

Default:

* timeframe: `1h`
* timerange: longer history, for example 12 months

### Balanced

Default:

* timeframe: `30m`
* timerange: medium history, for example 9–12 months

### Aggressive

Default:

* timeframe: `5m` or `15m`
* timerange: shorter but enough data, for example 3–6 months

Important:

* Do not force the timeframe if user manually overrides it.
* Show an “Auto” mode and a “Manual override” mode.

Suggested UI:

Timeframe:

* `Auto from risk profile`
* `Manual`

If Auto:

* show selected auto timeframe
* explain why it was selected

If Manual:

* user can select timeframe from dropdown

Supported timeframe options should be realistic for Freqtrade, for example:

* `1m`
* `3m`
* `5m`
* `15m`
* `30m`
* `1h`
* `4h`
* `1d`

Only include options the backend/Freqtrade setup can support.

---

## 4. Replace Days with Timerange

Remove the “days” field from the Optimization UI.

Replace it with timerange configuration.

The user should have:

### Timerange Auto

Auto-select timerange based on risk profile.

Examples:

* Lower Risk: last 12 months
* Balanced: last 9–12 months
* Aggressive: last 3–6 months

### Timerange Manual

User can select:

* start date
* end date

Then frontend/backend converts it into the Freqtrade timerange format if needed.

Example:

`20240101-20240630`

Rules:

* end date must be after start date
* show readable date range
* show resulting timerange string if useful
* do not allow invalid timerange silently

---

## 5. Remove “Download Missing Data” Checkbox

Remove the checkbox from the UI.

Missing data should be handled automatically.

Expected behavior:

* Before optimization, backend checks required data.
* If required data is missing, backend downloads it automatically.
* If download succeeds, optimization continues.
* If download fails, return controlled failure with a clear message.
* UI should show this as part of preflight/progress:

  * Checking data
  * Downloading missing data
  * Data ready
  * Optimization started

Do not require the user to decide this.

The user should not need a checkbox for something required.

Important:

* Do not fake data download.
* Do not mark data ready unless backend confirms it.
* Do not silently skip missing data.

---

Improve exchange selection.

Preferred approach:

* Use backend-supported exchange list if available.

Exchange selection should affect:

* optimization request
