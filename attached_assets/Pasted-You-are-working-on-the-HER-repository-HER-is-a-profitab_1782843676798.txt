You are working on the HER repository.

HER is a profitable-strategy discovery and validation system.

It tries to find profitable trading strategies, but it never claims profitability without evidence.

Core rule:

AI suggests.
Backend validates.
Freqtrade tests.
HER decides.

Your task is to redesign and improve the frontend UI/UX while preserving backend behavior.

Do not rebuild the backend.
Do not invent APIs.
Do not add fake data.
Do not add mock results.
Do not add approval/export/live trading.
Do not claim guaranteed profitability.

The backend is the source of truth.

---

# Main Goal

Create a beautiful, modern, workflow-driven frontend for HER.

The user should clearly understand:

1. Where they are in the strategy journey.
2. What has already happened.
3. What is currently running.
4. Whether the strategy is improving or failing.
5. What evidence supports the result.
6. What the next safe action is.

The UI must make HER feel like a real strategy discovery cockpit, not a collection of disconnected backend pages.

---

# Product Direction

HER should guide the user through this journey:

Strategy Workspace
→ Readiness Check
→ Baseline Run
→ Baseline Evidence
→ Optimization Run
→ Trial Progress
→ Best Trial / Optimized Candidate
→ Baseline vs Optimized Comparison
→ Validation Run
→ OOS / WFO / Robustness Evidence
→ Final Evidence Summary
→ Next Action

The UI should be designed around this journey.

---

# Design Style

Use a premium dark dashboard style.

Visual direction:

* clean trading lab interface
* dark background
* soft cards
* subtle borders
* modern sidebar
* compact but readable metric cards
* status badges
* step timeline
* live run progress
* charts for optimization and validation
* no clutter
* no fake “profit guaranteed” language

Recommended visual feel:

* dark navy / charcoal background
* glass-like cards but not too shiny
* accent color: electric blue or cyan
* success: controlled green
* warning: amber
* failure: red
* neutral: slate gray
* typography: clean, technical, readable
* spacing: generous enough to feel premium

Do not use loud casino/trading hype visuals.

HER must feel serious, analytical, and evidence-based.

---

# Required UX Principle

Every page must answer:

* What is this?
* What data is real?
* What is the current status?
* What evidence exists?
* What failed, if anything?
* What should the user do next?

No page should show raw backend data without explanation.

# Required Main Page: Strategy Journey

Create or improve a main guided workflow page.

Suggested route:

`/journey`

or integrate into the dashboard if the app already has a better main route.

This page should show one selected strategy and its full lifecycle:

## Sections

### 1. Strategy Header

Show:

* strategy name
* readiness status
* sidecar JSON status
* selected pairs
* timeframe
* risk profile
* latest run status

Actions:

* Open Strategy Workspace
* Run Baseline
* Run Optimization if baseline candidate exists
* Run Validation if baseline/optimized candidate exists

Use confirmation dialogs before any run action.

---

### 2. Journey Timeline

Create a horizontal or vertical stepper:

1. Strategy selected
2. Readiness checked
3. Baseline completed
4. Optimization completed
5. Validation completed
6. Candidate decision

Each step should have:

* status: not started / running / passed / failed / blocked
* short message
* timestamp if available
* link to detail page
* next action button where appropriate

This timeline must not fake completion.

If no backend data exists, show “Not started”.

---

### 3. Live Run Workflow Panel

When a run is active, show a live workflow panel.

This panel should show:

* current stage
* completed stages
* running stage
* failed stage if any
* elapsed time if available
* last updated time
* controlled failure message if failed
* progress percentage only if it can be derived safely

Use polling if WebSocket is not available.

Suggested polling:

* every 2 seconds while status is running
* stop polling when completed / failed / blocked
* do not overload backend

Do not invent live progress if backend does not provide it.

If progress percentage is unknown, show stage-based progress instead.

# Required User Flow

After your changes, this flow should work from the UI:

1. User opens Strategy Workspace.
2. User selects a ready strategy.
3. User starts baseline with confirmation.
4. User sees baseline run progress.
5. User opens baseline detail.
6. User sees metrics, decision, pair results, and next action.
7. User can start optimization if baseline is candidate/promising.
8. User sees optimization live progress and trial charts.
9. User opens optimization detail.
10. User sees best trial and baseline-vs-optimized comparison.
11. User can start validation from baseline or optimization detail.
12. User confirms validation.
13. User is redirected to validation detail.
14. User sees OOS, WFO, robustness, final decision, and warnings.
15. User sees next safe action.

If a step cannot be completed because backend support is missing, document the exact missing endpoint or field. Do not fake it.
