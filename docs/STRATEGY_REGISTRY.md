# Strategy Registry

## Purpose

The Strategy Registry is a database-backed system for managing trading strategies and their versions. It provides a centralized way to track strategies generated by AI, uploaded by users, repaired from existing strategies, or manually created. The registry separates strategy metadata from actual strategy code files, allowing for versioning and history tracking without file system dependencies.

## Strategy vs Strategy Version

### Strategy
A **Strategy** represents a logical trading strategy concept. It contains metadata about the strategy:
- Name and class name
- Source type (how it was created)
- Direction (long, short, both, unknown)
- Timeframe
- Status (draft, active, archived)
- Current version reference

### Strategy Version
A **Strategy Version** represents a specific implementation of a strategy. Each version contains:
- Version number (auto-incremented)
- File paths to `.py` and `.json` sidecar files
- Strategy specification (parameters, indicators, etc.)
- Strategy parameters (configuration values)
- Code hash (for integrity verification)
- Creation context (which run created it)
- Notes

**Key Concept:** One Strategy can have multiple Versions. Only one version is marked as "current" at any time.

## Strategy Source Types

Strategies can be created through different methods:

- **generated** - Created by AI through the AutoQuant pipeline
- **uploaded** - Uploaded by the user from an existing file
- **repaired** - Repaired from a failed or broken strategy
- **manual** - Manually created by the user
- **imported** - Imported from another HER-compatible source
- **demo** - Safe placeholder strategy data for frontend/API development

Each source type is tracked to understand the origin of the strategy.

## Strategy File Locations

Strategy files will be stored in the following locations (in future parts):

```
freqtrade_workspace/user_data/strategies/
├── {strategy_id}_v{version_number}.py    # Strategy Python code
└── {strategy_id}_v{version_number}.json   # Strategy parameters sidecar
```

The file naming convention includes the strategy ID and version number to ensure uniqueness and easy version tracking.

## .py and .json Sidecar Rule

Every strategy version consists of two files:

1. **`.py` file** - The actual Freqtrade strategy Python code
2. **`.json` sidecar file** - Strategy parameters and metadata

This separation allows:
- Easy parameter modification without touching code
- Clear separation between logic and configuration
- Better version control and diffing
- Freqtrade compatibility (Freqtrade uses this pattern)

The JSON sidecar contains:
- Strategy parameters (stoploss, roi, timeframe, etc.)
- Indicator parameters
- Any configuration needed by the strategy

## Current Version

Each strategy has a `current_version_id` field that points to the currently active version. This is used by:
- The frontend to display the active strategy
- Freqtrade to know which version to backtest
- The export system to know which version to export

When a new version is created:
- If it's version 1, it automatically becomes current
- Otherwise, the current version must be explicitly set via API

## Tracking Generated/Uploaded/Repaired Strategies

### Generated Strategies
- Created by AI during the AutoQuant pipeline
- `source_type = "generated"`
- `created_from_run_id` links to the run that generated it
- Version 1 is automatically created when the strategy is first generated

### Uploaded Strategies
- Uploaded by the user from an existing file
- `source_type = "uploaded"`
- File path is set to the upload location
- Initial version is created from the uploaded file

### Repaired Strategies
- Created when a strategy fails and needs fixing
- `source_type = "repaired"`
- `parent_run_id` links to the original run
- Can be based on an existing strategy version

### Imported Strategies
- Imported from another HER-compatible source
- `source_type = "imported"`
- Must still be validated before any trading use

### Demo Strategies
- Created by `scripts/seed-demo-data.py` for local development
- `source_type = "demo"`
- Always marked `is_demo = true`
- Must never be treated as real trading evidence

## No Free-Form AI Code Rule

**Important:** The Strategy Registry does NOT generate free-form AI code. All strategy generation is controlled and structured:

1. **Structured Prompts** - AI receives structured prompts with specific requirements
2. **Validation** - Generated code is validated against Freqtrade requirements
3. **Testing** - Strategies are backtested before being accepted
4. **Versioning** - Each generated version is tracked with full context
5. **Audit Trail** - All AI interactions are logged in the audit system

The registry stores metadata and references to files, but does not execute AI generation itself. AI generation happens in a separate pipeline with strict controls.

## Strategy Status Lifecycle

Strategies move through these statuses:

1. **draft** - Initial state, not yet validated
2. **active** - Validated and ready for use
3. **archived** - No longer in use but kept for history

Strategies can be archived to remove them from active lists while preserving their history.

## Demo Strategies

Demo strategies are marked with `is_demo = true`. These are:
- Used for testing and development
- Can be deleted in bulk via `delete_demo_strategies()`
- Kept separate from user strategies
- Useful for onboarding and examples

## API Endpoints

The Strategy Registry provides REST API endpoints for:
- Creating, reading, updating strategies
- Archiving strategies
- Managing versions
- Setting current version
- Listing strategies with filters

See `docs/API_CONTRACTS.md` for detailed API documentation.
